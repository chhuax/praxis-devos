本规范旨在提供通用的 Redis 使用指导，适用于所有技术栈，确保 Redis 集群的高性能、高可用与稳定性。

## 1. 禁止与高危命令

在生产环境中，严禁执行可能导致 Redis 阻塞或数据丢失的操作。

```text
❌ KEYS *          -> 线上严禁使用，会导致全库扫描并阻塞服务。替代方案：SCAN
❌ FLUSHDB/FLUSHALL -> 严禁非运维操作。清空数据会导致缓存雪崩。
❌ MONITOR          -> 高并发下会导致 Redis 性能剧烈下降。
❌ SAVE             -> 阻塞操作，应使用 BGSAVE。
❌ CONFIG           -> 生产环境应通过配置文件或运维管理，严禁在应用中调用。
❌ DEBUG            -> 仅用于本地调试。
```

---

## 2. 命令复杂度意识

理解命令的时间复杂度是保证 Redis 高性能的关键。

### 复杂度评估

1. **优先使用 O(1) 命令**：如 `GET`, `SET`, `HGET`, `LPOP` 等。
2. **慎用 O(N) 命令**：
   - 集合操作（`SMEMBERS`, `HGETALL`, `LRANGE`）在元素较多时会阻塞。
   - 替代方案：使用 `SSCAN`, `HSCAN`, `ZSCAN` 分页获取。
3. **批量删除**：
   - ❌ 直接 `DEL` 大集合。
   - ✅ 使用 `UNLINK`（Redis 4.0+）进行异步删除。

### 批量操作规范

- **Pipeline**：将多个命令打包发送，减少 RTT（往返时延）。建议每次 Pipeline 的 Key 数量 **≤ 100**。
- **MSET/MGET**：适用于简单的批量读写。

## 3. Key 设计原则

合理的 Key 设计便于维护并能节省内存。

### 命名规范

1. **使用冒号分层（Namespace）**：
   - 格式：`业务名:模块名:唯一标识`
   - ✅ `order:detail:12345`
2. **简洁且有意义**：
   - 在保证语义清晰的前提下，缩短 Key 长度（减少内存消耗）。
   - ❌ `application_user_authentication_token:123`
   - ✅ `auth:token:123`
3. **字符限制**：
   - 仅限英文字母、数字、点(.)、下划线(_)和冒号(:)。
   - 禁止使用空格、中文、换行符或特殊转义字符。

---

## 4. Value 大小建议

避免 "BigKey" 是 Redis 优化的重中之重。

| 类型 | 推荐阈值 | 建议 |
|------|---------|------|
| **String** | ≤ 10 KB | 严禁超过 1 MB。大数据建议压缩存储（如 GZIP/Snappy）。 |
| **Hash** | 元素 < 500 | 字段过多会增加内存碎片。 |
| **List/Set/ZSet** | 元素 < 5000 | 严禁存储百万级别的超大集合，查询性能会严重下降。 |

---

## 5. 过期与缓存抖动

未设置过期时间是 Redis 内存溢出的首要原因。

1. **必须设置过期时间**：除非是基础配置类数据，所有 Key 理论上都应有过期限制。
2. **分散过期时间（Jitter）**：
   - 为避免大量 Key 在同一时刻过期（导致缓存雪崩），在设置 TTL 时增加随机扰动。
   - **示例（伪代码）**：
     ```text
     base_ttl = 3600
     random_jitter = random(-300, 300)
     SETEX key (base_ttl + random_jitter) value
     ```

---

## 6. 连接与批处理

1. **使用连接池**：严禁频繁创建和销毁连接（短连接）。应配置合理的 `maxActive`, `maxIdle`, `minIdle`。
2. **超时配置**：设置合理的连接超时（Connection Timeout）和读写超时（SoTimeout），防止连接池被耗尽。

---

## 7. 缓存交互模式

根据业务需求选择合适的缓存模式：

- **Cache-Aside（最通用）**：
  - 读：先读缓存，不中则读数据库并写缓存。
  - 写：先写数据库，然后 **删除** 缓存（推荐）。
- **Read/Write-Through**：应用只与缓存交互，由缓存服务负责同步数据库。
- **Write-Behind（异步写）**：先写缓存，由后台异步批量写入数据库。适用于写并发极高且允许短暂不一致的场景。

---

## 8. 缓存典型问题

| 问题 | 定义 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据，穿透到 DB | 1. 缓存空值（TTL 短）<br>2. 布隆过滤器 (Bloom Filter) |
| **缓存击穿** | 热点 Key 过期瞬间，海量请求压垮 DB | 1. 互斥锁（SET NX）<br>2. 逻辑过期（后台异步更新） |
| **缓存雪崩** | 大量 Key 同时过期或 Redis 宕机 | 1. 分散过期时间<br>2. 多级缓存<br>3. 限流熔断 |

## 9. 分布式锁规范

1. **互斥性**：使用 `SET key value NX PX milliseconds` 确保原子性加锁。
2. **释放锁安全性**：使用 **Lua 脚本** 校验 Value 是否匹配，防止误删他人的锁。
   - **Lua 示例**：
     ```lua
     if redis.call("get", KEYS[1]) == ARGV[1] then
         return redis.call("del", KEYS[1])
     else
         return 0
     end
     ```
3. **避免长耗时**：锁持有的业务逻辑应尽量简短。如需长锁，考虑红锁（Redlock）或增加看门狗（Watchdog）自动续期机制。

## 10. 数据结构选型

| 结构 | 特点 | 适用场景 |
|------|------|---------|
| **String** | 简单 KV，计数器 | 共享 Session、分布式锁、普通对象缓存 |
| **Hash** | 结构化存储，节省内存 | 存储对象（如用户信息）、商品库存 |
| **List** | 双向列表，有序，允许重复 | 简单队列、最新消息排行 |
| **Set** | 无序，去重，支持交并集 | 共同关注、标签系统、唯一计数 |
| **ZSet** | 有序，按权重排序 | 实时排行榜、带优先级的任务队列 |

## 11. 淘汰策略建议

在内存触达上限时的处理策略：

- **volatile-lru (推荐)**：仅从设置了过期时间的 Key 中按 LRU 淘汰，保护永久数据。
- **allkeys-lru**：从所有 Key 中按 LRU 淘汰，适用于将 Redis 纯作为缓存使用的场景。
- **noeviction**：默认策略。写满报错，不建议生产使用。
