# 代码评审技能 (Code Review)

## 目的

代码评审是保障项目代码质量的核心环节，旨在：
1. **保障质量**：在代码合并前发现潜在的 Bug、性能瓶颈与安全风险。
2. **知识传递**：促进团队成员间的设计思路交流与技术方案同步。
3. **维护一致性**：确保代码符合技术栈规范（Stack Rules）与项目架构要求。
4. **规范闭环**：验证代码实现与 OpenSpec 规范场景（Scenarios）的完全匹配。

## 何时使用此技能

- **自审阶段**：在提交代码或发起 Pull Request (PR) 前进行最后的质量检查。
- **评审阶段**：作为评审者对他人的代码变更进行审查。
- **修复阶段**：根据评审反馈进行代码优化与缺陷修复。
- **合并决策**：评估 PR 是否满足合并标准。

## 评审清单 (Checklist)

### 1. 正确性 (Correctness)
- [ ] 代码是否完整实现了需求？是否遗漏了 `openspec` 中的任何场景？
- [ ] 逻辑是否存在缺陷（如：Off-by-one errors、空指针风险、资源未释放）？
- [ ] 异常处理是否完备？是否考虑了所有可能的失败路径？

### 2. 安全性 (Security)
- [ ] 是否存在敏感信息泄露（密码、密钥、个人隐私数据）？
- [ ] 是否对所有外部输入进行了有效校验（防注入、防溢出）？
- [ ] 是否遵循了最小权限原则？

### 3. 性能 (Performance)
- [ ] 是否存在明显的性能隐患（如：循环内查询数据库、不必要的对象创建）？
- [ ] 算法复杂度是否合理？是否使用了合适的并发处理机制？
- [ ] 资源使用是否高效（内存、CPU、网络带宽）？

### 4. 可读性与可维护性 (Readability & Maintainability)
- [ ] 变量与函数命名是否直观、无歧义？
- [ ] 函数是否过长？逻辑嵌套是否过深（建议不超过 3 层）？
- [ ] 是否存在重复代码（DRY 原则）？
- [ ] 注释是否清晰？是否解释了“为什么”而不是简单的“是什么”？

### 5. 测试覆盖 (Test Coverage)
- [ ] 是否包含必要的单元测试或集成测试？
- [ ] 测试是否覆盖了核心逻辑、异常路径与边界情况？
- [ ] 测试用例是否独立且可重复运行？

### 6. 边界条件 (Edge Cases)
- [ ] 集合为空、数值溢出、超时、网络中断等异常情况是否有应对方案？
- [ ] 是否处理了并发竞争条件（Race Conditions）？

## PR/MR 描述规范

一个高质量的 PR 描述是高效评审的前提，应包含：
1. **摘要 (Summary)**：用 1-2 句话描述变更的核心目的。
2. **变更类型**：如 `feat`, `fix`, `refactor`, `perf` 等。
3. **关联变更 (Links)**：关联的 `openspec` 提案 ID 或 Issue 链接。
4. **关键变更点**：列出核心修改的文件或逻辑改动。
5. **验证步骤**：说明评审者如何运行测试或手动验证该变更。
6. **风险提示**：是否有破坏性变更（Breaking Changes）或潜在的副作用。

## 评审反馈分级

评审者应根据问题的严重程度标记反馈等级：

| 等级 | 标识 | 说明与行动 |
| :--- | :--- | :--- |
| **建议** | 💡 Suggestion | 针对代码改进的小建议或风格优化，不强制修改，可直接合并。 |
| **需要修改** | ⚠️ Request Changes | 存在明显的逻辑缺陷、命名误导或测试不足，**必须**修复后才能合并。 |
| **阻塞** | 🚫 Blocker | 存在重大安全漏洞、架构错误或严重性能问题，**严禁**合并，需重新设计。 |

## 自审流程 (Self-Review)

在邀请他人评审前，请务必执行以下步骤：
1. **对比规范**：再次阅读 `openspec/changes/<change-id>/` 下的提案，确保每个 Scenario 都有对应的代码实现与测试验证。
2. **运行验证**：本地运行全量测试，确保无回归问题。
3. **清理代码**：删除调试日志（console.log, print）、临时注释与未使用的导入。
4. **审查 Diff**：逐行阅读自己的 `git diff`，确保每一行变更都是必要的且符合规范。
5. **证据先于断言**：在 PR 中附上测试通过的截图或日志作为实现成功的证据。

## 评审者行为准则

- **对事不对人**：评审的目标是代码质量，而非开发者个人。避免使用指责性语言。
- **多问少定论**：使用“这里是否考虑过...？”而非“这里你应该用...”。
- **提供替代方案**：指出问题时，尽量给出一个更好的实现示例或思路。
- **及时反馈**：尽量在 24 小时内给出初步评审结果，避免 PR 长期堆积。
- **赞美闪光点**：如果看到优雅的实现或精妙的代码，不要吝惜你的称赞。

## 核心规则

### ✅ 应该做
- **坚持验证优先**：按风险选择测试策略，并为关键结论提供可重复验证证据。
- **小步提交**：保持 PR 的粒度适中，一个 PR 只关注一个原子变更。
- **遵循技术栈规则**：严格执行 `stacks/{stack}/rules.md` 中的编码规范。
- **关联规范上下文**：如果存在 OpenSpec `change-id`，在 PR 中显式关联它。
- **基于证据沟通**：引用具体的性能指标或规范文档来支撑你的评审建议。

### ❌ 不应该做
- **禁止 执行仪式化评审**：不要只看代码格式而忽略核心逻辑。
- **禁止 盲目妥协**：评审不应是表演性质的附和，发现问题必须客观指出。
- **禁止 引入无关变更**：不要在修复 Bug 的 PR 中顺便重构无关的模块。
- **禁止 在评审中争论不休**：如果对方案有重大分歧，应加载 `brainstorming` 技能或升级至架构评审（Oracle）。

## 协作纽带

本技能与以下技能共同构成开发闭环：
- `openspec`：提供 **规范依据**（评审代码是否符合需求）。
- `git-workflow`：提供 **协作通道**（如何发起、更新与合并 PR）。
- `code-review`：提供 **质量保障**（确保最终入库的代码是高质量的）。
