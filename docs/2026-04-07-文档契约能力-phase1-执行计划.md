# 文档契约能力 Phase 1 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 为 `praxis-devos` 落地文档契约能力的第一阶段，实现 `--docs-contracts` 模式、`docs init`、`docs check`、托管规则注入，以及最小测试覆盖。

**Architecture:** 这一阶段只做“结构、门禁、校验、初始化”四类能力，不做复杂的自动文档生成和代码 diff 推断。CLI 入口统一收敛到 [`src/core/praxis-devos.js`](/Users/huaxin/Documents/workspace/praxis-devos/src/core/praxis-devos.js)，模板资产放在仓库内，OpenSpec 和 AGENTS 注入继续沿用现有 managed block 机制。

**Tech Stack:** Node.js ESM、内置 `node:test`、文件系统模板同步、OpenSpec managed entry、Praxis 现有 CLI 分发。

## 1. 已锁定决策

- 文档契约能力采用项目级模式开关：`off / minimal / full`
- 默认模式是 `minimal`

- Praxis 不替用户写完整文档；Praxis 负责安装骨架、注入门禁、提供检查和最小同步能力
- 中文模板当前已作为产品内置模板目标资产存在于 [`docs/harness-engineering-template-zh/`](/Users/huaxin/Documents/workspace/praxis-devos/docs/harness-engineering-template-zh)

## 5. 设计注意事项

### 5.1 不要在 Phase 1 过度设计

本阶段不要做：

- 复杂 DSL 级别的 `catalog.yaml` 解析器
- 自动扫描代码推断 `contract impact`
- 自动生成长篇 guide
- 自动修改 OpenSpec skill 资产

### 5.2 模板源不要双向漂移

需要明确：

- `src/templates/docs-contracts/` 才是产品内置模板
- `docs/harness-engineering-template-zh/` 是说明和示例，还是产品源资产

建议 Phase 1 结束前统一成单一源，不要维护两套长期分叉模板。

### 5.3 测试风格

沿用现有 `node:test` 风格，不引入新测试框架。
