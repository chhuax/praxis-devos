# 文档契约能力可实施规格

## 1. 当前目标

当前阶段的目标不是一次性落完整文档治理体系，而是先解决一个更直接的问题：

**让 AI 在进入用户项目后，能够快速掌握项目，并知道项目最重要的对外面在哪里。**

基于这个目标，第一阶段只落两类能力：

- `docs/codemaps/`
- `docs/surfaces.yaml`

## 2. 为什么收缩范围

此前讨论过更完整的文档契约体系，例如：

- `contracts/` 下拆多类契约
- `docs/reference/`
- `docs/guides/`
- `docs/runbooks/`
- `checks` / `definition-of-done`
- OpenSpec 各阶段 docs gate

这些方向本身是成立的，但第一步过重，会带来两个问题：

1. 对用户项目接入成本过高
2. 对 AI 快速掌握项目的直接收益不够集中

因此第一阶段应收缩成：

**先解决“快速上手”和“主对外面定位”，再考虑完整治理。**

## 3. 第一阶段范围

### In Scope

- 为用户项目安装 `docs/codemaps/` 模板
- 为用户项目安装 `docs/surfaces.yaml` 模板
- 在托管规则中提示 AI 优先查看 codemap 和 surfaces
- 提供最小初始化能力
- 提供最小存在性校验能力

### Out Of Scope

- 细分 HTTP / SDK / CLI / events / config 契约目录
- `docs/reference/` / `docs/guides/` / `docs/runbooks/` 全套模板
- `off / minimal / full` 分级治理
- OpenSpec propose / apply / archive 的文档 gate
- 自动 diff 推断文档影响
- 自动生成完整 API 文档

## 4. 核心结构

第一阶段只要求用户项目具备下面两类目录或文件：

```text
docs/
  codemaps/

contracts/
  surfaces.yaml
```

## 5. `docs/codemaps/` 规格

### 5.1 目标

`codemaps` 是给 AI 快速消费的项目地图，不是传统架构文档。

它的主要作用是帮助 AI 快速回答：

- 这是个什么项目
- 主入口在哪
- 核心模块在哪
- 哪类问题应该先看哪些文件
- 当前任务可能落在哪些目录

### 5.2 内容要求

每份 codemap 应尽量短、路径明确、少写实现细节。

建议包含：

- 一句话概述
- 首先看哪些文件
- 仓库结构或模块结构速览
- 问题路由
- 关键事实

### 5.3 第一阶段推荐文件

至少安装：

- `docs/codemaps/project-overview.md`

可选按项目类型扩展：

- `docs/codemaps/api.md`
- `docs/codemaps/sdk.md`
- `docs/codemaps/cli.md`
- `docs/codemaps/deployment.md`

Praxis 自身仓库当前采用的是多文件 codemap 方案，这只是本仓库示例，不强制用户项目完全照搬。

## 6. `docs/surfaces.yaml` 规格

### 6.1 目标

`surfaces.yaml` 是用户项目“主对外面”的统一入口。

它不要求一上来把所有接口都做成详细契约，只要求先回答：

- 这个项目有哪些对外面
- 哪个是主对外面
- 主对外面的事实源位置在哪里
- 该对外面属于哪种类型

### 6.2 为什么先做一个文件

用户项目的对外面不一定是 HTTP API，也可能是：

- SDK
- CLI
- 事件
- 配置面

如果第一阶段直接拆出很多目录：

- `http/`
- `sdk/`
- `cli/`
- `events/`
- `config/`

会让接入过重，也不利于先统一认知。

因此第一阶段只保留：

- `docs/surfaces.yaml`

后续如果某个项目成熟，再从 `surfaces.yaml` 扩展出更细的契约文件。

### 6.3 示例

```yaml
primary_surface: public-sdk

surfaces:
  - id: public-sdk
    kind: sdk
    location: src/index.ts
    contract_ref: docs/reference/sdk-public.md
    description: 对外导出的 SDK 入口

  - id: project-config
    kind: config
    location: config/schema.json
    contract_ref: docs/reference/config.md
    description: 项目运行时配置面
```

### 6.4 字段说明

- `primary_surface`
  - 当前项目最重要的主对外面
- `surfaces[].id`
  - 对外面唯一标识
- `surfaces[].kind`
  - 对外面类型，第一阶段只做标签用途
- `surfaces[].location`
  - 真实事实源或主入口位置
- `surfaces[].contract_ref`
  - 人类可读文档引用，可选
- `surfaces[].description`
  - 一句话说明

### 6.5 第一阶段支持的 `kind`

第一阶段不做强 schema 校验，只建议使用以下枚举：

- `http`
- `sdk`
- `cli`
- `events`
- `config`
- `other`

## 7. CLI 规格

第一阶段建议新增最小命令：

### 7.1 `setup`

在现有 `setup` 上增加轻量文档结构初始化能力。

行为：

- 补齐 `docs/codemaps/`
- 补齐 `docs/surfaces.yaml`
- 不覆盖用户现有文件

### 7.2 `sync`

行为：

- 刷新托管规则中的 codemap / surfaces 提示
- 补齐缺失的 codemap 模板和 `surfaces.yaml`

### 7.3 `docs init`

建议支持：

```bash
npx praxis-devos docs init
```

行为：

- 仅初始化 `docs/codemaps/` 和 `docs/surfaces.yaml`

### 7.4 `docs check`

建议支持：

```bash
npx praxis-devos docs check
```

第一阶段只检查：

- `docs/codemaps/project-overview.md` 是否存在
- `docs/surfaces.yaml` 是否存在
- `surfaces.yaml` 是否声明 `primary_surface`
- `primary_surface` 是否能在 `surfaces` 列表中找到
- `location` 是否填写
- 是否残留明显占位符

第一阶段 `docs check` 只做提示性输出，不做强 gate。

修改 [`src/templates/managed-entry.md`](/Users/huaxin/Documents/workspace/praxis-devos/src/templates/managed-entry.md)，增加轻量提示：

## 9. docs 子 agent 协作模式

第一阶段不在 Praxis 内部实现复杂的 docs 编排器，但要把协作边界设计清楚，便于安装到用户项目后由宿主 agent 采用。

### 9.1 角色分工

- 主 agent
  - 判断当前变化是否影响主对外面
  - 更新 `docs/surfaces.yaml`
  - 决定是否需要刷新 `docs/codemaps/`
  - 最终执行 `docs check`
- docs 子 agent
  - 阅读 `docs/surfaces.yaml`
  - 阅读已有 `docs/codemaps/`
  - 刷新 `docs/codemaps/**`
  - 不改代码
  - 不改 `docs/surfaces.yaml`
  - 不决定任务是否整体完成

### 9.2 为什么这样设计

这样做是为了把“事实源更新”和“派生文档刷新”拆开：

- `surfaces.yaml` 更接近真实性判断，应由主 agent 负责
- `codemaps` 更适合并行 sidecar 更新，应由 docs 子 agent 负责
- 主 agent 最后统一验收，避免子 agent 自己宣称完成

这一模式借鉴 ECC 的 `doc-updater` 思路，但只保留最小边界，不引入额外运行时系统。

### 9.3 第一阶段如何落地

第一阶段只做“可协作骨架”，不做自动派发：

- Praxis 负责把 `docs/codemaps/` 和 `docs/surfaces.yaml` 安装进用户项目
- 托管规则中提示：
  - 主 agent 先更新 `surfaces.yaml`
  - docs 更新可交由子 agent 处理
  - docs 子 agent 只应修改 `docs/codemaps/**`
- `docs check` 仍由主 agent 在收尾前执行

### 9.4 后续可扩展项

后续阶段可再补：

- `praxis-devos docs prepare --change <name>`
- `openspec/changes/<change>/docs-handoff.yaml`
- 更明确的 docs 子 agent 输入输出约定

这些能力不属于第一阶段范围。

## 10. AI 使用方式

第一阶段的 AI 消费路径应是：

1. 先看 `AGENTS.md`
2. 再看 `docs/codemaps/project-overview.md`
3. 再看 `docs/surfaces.yaml`
4. 根据 `primary_surface` 决定继续读哪个入口文件或哪个契约引用

这样 AI 至少能快速建立：

- 项目地图
- 主对外面
- 事实源入口

## 11. 后续升级路径

第一阶段完成后，后续可逐步增加：

### Phase 2

- 按项目类型扩展更多 codemap
- 让 `surfaces.yaml` 支持更多字段
- 增加轻量 `contract_ref` 校验

### Phase 3

- 为特定 `kind` 扩展细分契约目录
- 引入 `reference / guides / runbooks`
- 引入更强的 checks / DoD
- 再与 OpenSpec gate 深度集成

## 12. 最终结论

第一阶段不追求“完整文档治理”，只追求：

- **AI 快速掌握项目**
- **快速定位主对外面**

因此最合适的最小实现是：

```text
docs/codemaps/
docs/surfaces.yaml
```

这是后续完整文档契约体系的最小起点，而不是最终形态。
