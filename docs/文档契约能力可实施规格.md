# 文档契约能力可实施规格

## 1. 目标

为 `praxis-devos` 增加一套面向 AI 协作开发的文档契约能力，满足以下目标：

- 对外行为先落契约，再落参考文档
- 文档同步进入 OpenSpec 变更流程，而不是依赖人工记忆
- 用户项目可按成熟度选择治理强度
- 文档和契约能够作为大模型的稳定上下文来源

设计原则：

- 按 `3` 设计：考虑运行时变更感知、文档建议、上下文反哺
- 按 `2` 落地：第一阶段只交付模板、CLI、OpenSpec gate、docs check

## 2. 范围

第一阶段落地范围：

- `setup` / `sync` 支持文档契约模式
- 初始化 `contracts/` 与 `docs/` 最小骨架
- 在托管规则中注入 `Documentation / Contract Gate`
- 在 OpenSpec propose / apply / archive 阶段接入文档影响判断
- 提供 `docs init`、`docs plan`、`docs check`、`docs sync`

第一阶段不做：

- 自动识别所有代码差异并推断所有文档影响
- 自动生成完整 guide / runbook 内容
- 自动把全部上下文打包进模型提示词

## 3. 模式设计

文档契约能力采用项目级策略开关，不采用大量细粒度开关。

### 3.1 模式枚举

- `off`
- `minimal`
- `full`

### 3.2 模式语义

#### `off`

适用场景：

- 一次性脚本
- 非长期维护的小工具
- 用户明确不希望引入文档契约治理

行为：

- 不初始化 `contracts/`
- 不初始化 `docs/` 模板
- 不注入 docs gate
- 不在 completion / archive 前要求 `docs check`

#### `minimal`

适用场景：

- 0 文档项目
- 迁移早期项目
- 想先建立文档骨架但不希望硬阻塞的项目

行为：

- 初始化最小契约和文档骨架
- 注入 docs gate，但作为建议模式
- `docs check` 提供告警，不阻塞 completion / archive
- OpenSpec change 中记录 doc impact，但允许为空并带说明

#### `full`

适用场景：

- 平台类项目
- 对外服务或接口项目
- 企业项目
- 已有较成熟 OpenSpec 流程的项目

行为：

- 初始化完整契约和文档骨架
- docs gate 作为硬约束
- `docs check` 失败会阻止 completion / archive，除非显式豁免
- breaking change 必须补 migration
- OpenSpec 变更必须记录 contract impact / doc impact

### 3.3 默认策略

默认模式：`minimal`

原因：

- `off` 会让框架失去统一性
- `full` 对低成熟度项目接入成本过高
- `minimal` 既能建立结构，又不会一上来变成硬门禁

### 3.4 升级建议条件

满足任一条件时，`setup` 或 `doctor` 可以提示用户考虑升级到 `full`：

- 项目存在 `openspec/`
- 项目存在 `docs/`
- 存在 controller / route / api 目录
- 检测到 OpenAPI / Swagger / Spring MVC / Nest / Express 路由
- 用户选择多 agent 项目
- 用户启用了企业规则包

## 4. CLI 规格

### 4.1 `setup`

新增参数：

```bash
npx praxis-devos setup --docs-contracts off
npx praxis-devos setup --docs-contracts minimal
npx praxis-devos setup --docs-contracts full
```

语义：

- `--docs-contracts` 不传时默认 `minimal`
- 若项目已存在相关目录，只补缺失文件，不覆盖用户现有内容

### 4.2 `sync`

新增行为：

- 刷新托管规则中的 docs gate
- 若启用 `minimal` 或 `full`，补齐缺失的模板文件
- 不覆盖用户已经修改的正式文档

### 4.3 `docs init`

命令：

```bash
npx praxis-devos docs init --mode minimal
npx praxis-devos docs init --mode full
```

用途：

- 单独初始化文档契约骨架
- 适用于已经完成 `setup` 的老项目补装能力

### 4.4 `docs plan`

命令：

```bash
npx praxis-devos docs plan --change <name>
```

输出：

- 当前 change 涉及的契约面
- 需要同步的 `reference`
- 可能需要补的 `guides`
- 可能需要补的 `runbooks`

第一阶段实现方式：

- 基于 `docs/catalog.yaml`
- 基于 change 内声明的 `contract impact` / `doc impact`
- 不做复杂代码 diff 推断

### 4.5 `docs check`

命令：

```bash
npx praxis-devos docs check
npx praxis-devos docs check --change <name>
npx praxis-devos docs check --json
```

第一阶段检查项：

- 契约文件是否存在
- `reference` 是否存在
- 是否存在占位符
- breaking change 是否补 migration
- runbook 变化是否补相应模板字段
- `catalog.yaml` 中声明的 contract / output 是否存在

模式差异：

- `minimal`：失败返回 warning 级结果
- `full`：失败返回 non-zero exit code

### 4.6 `docs sync`

命令：

```bash
npx praxis-devos docs sync
npx praxis-devos docs sync --change <name>
```

第一阶段职责：

- 执行最小 reference 同步
- 刷新从契约派生的基础标题或索引
- 不尝试自动生成复杂教程

说明：

第一阶段的 `docs sync` 应偏保守，避免伪智能生成大量低质量文档。

## 5. 初始化文件规格

### 5.1 `minimal` 初始化内容

创建以下目录或文件（仅在不存在时）：

```text
contracts/
  openapi/public.yaml
  config/app.schema.json
  errors/errors.yaml
  permissions/permissions.yaml
  cli/cli.yaml

docs/
  README.md
  catalog.yaml
  reference/api.md
  reference/config.md
  reference/errors.md
  reference/permissions.md
  guides/getting-started.md
  guides/integration-guide.md
  guides/contributing.md
  guides/migration-template.md
  runbooks/incident-template.md
  runbooks/api-failure-template.md
  adr/0001-template.md
  checks.md
  definition-of-done.md
```

### 5.2 `full` 初始化内容

与 `minimal` 相同，但增加：

- 托管规则中的硬 gate 提示
- OpenSpec 变更期文档检查要求
- 默认启用 completion / archive 前 `docs check`

## 6. 托管规则修改

修改 [`src/templates/managed-entry.md`](/Users/huaxin/Documents/workspace/praxis-devos/src/templates/managed-entry.md)，新增 `Documentation / Contract Gate`。

### 6.1 `minimal` 规则

- 对外 API、配置、权限、错误码、CLI 变化应优先更新契约
- 接入方式变化应同步 `guides`
- 运维恢复流程变化应同步 `runbooks`
- completion / archive 前建议执行 `docs check`

### 6.2 `full` 规则

- 对外 API、配置、权限、错误码、CLI 变化必须先更新契约
- breaking change 必须补 migration
- completion / archive 前必须通过 `docs check` 或显式豁免
- 不允许宣称完成但文档契约未同步

## 7. OpenSpec 集成规格

### 7.1 `opsx-propose`

Proposal 阶段新增以下结构化信息：

- `contract impact`
- `doc impact`
- `breaking change`
- `migration required`
- `affected surfaces`

示例：

```text
contract impact:
- public-http-api
- permissions

doc impact:
- reference/api
- guides/migration

breaking change: yes
migration required: yes
```

### 7.2 `opsx-apply`

Apply 阶段任务模板新增：

- 更新契约
- 同步 reference
- 如有必要，补 guide / migration
- 如有必要，补 runbook

要求：

- 如果 change 声明了 `contract impact`，则 tasks 中不能缺失对应文档任务

### 7.3 `opsx-archive`

Archive 阶段新增 `docs sync assessment`：

- 相关契约是否已更新
- `reference` 是否已同步
- `migration` 是否已补
- 是否存在占位符
- 是否允许带风险归档

模式差异：

- `minimal`：允许告警后归档
- `full`：默认阻止归档，除非明确豁免

## 8. `docs/catalog.yaml` 规格

`catalog.yaml` 是文档契约规则中心。

### 8.1 最小字段

- `id`
- `kind`
- `contract`
- `outputs`
- `detect`
- `triggers`
- `checks`

### 8.2 可执行字段

建议第一阶段支持：

- `detect.contractPaths`
- `detect.sourcePaths`
- `detect.changeKinds`
- `checks.contractRequired`
- `checks.referenceRequired`
- `checks.placeholderCheck`
- `checks.migrationRequiredWhen`

### 8.3 关联字段

建议支持轻量 `links`：

- API -> errors
- API -> permissions
- CLI -> config

目的：

- 用于影响分析
- 用于 docs plan
- 用于后续 context packing

## 9. 现场运行机制

这里的“安装到用户现场，如何在合适的时机去更新合适的文档”，第一阶段按以下机制落地。

### 9.1 安装时

`setup` 或 `docs init`：

- 创建骨架
- 注入托管规则
- 告知当前模式

### 9.2 提案时

`opsx-propose`：

- 让 agent 在 proposal 中声明本次变化影响哪些契约面

### 9.3 实现时

`opsx-apply`：

- 将契约同步和文档同步作为任务的一部分
- `docs plan` 可辅助列出应更新项

### 9.4 收尾时

`opsx-archive` / completion：

- 运行 `docs check`
- 根据模式决定是 warning 还是 hard gate

这意味着更新时机不是“代码写完再想起来”，而是：

`proposal 声明 -> apply 执行 -> archive 校验`

## 10. 如何反哺给 AI 大模型

这是设计按 `3` 考虑但首期按 `2` 落地时必须留好的接口。

### 10.1 第一阶段可直接反哺的内容

AI 通过以下稳定入口获取上下文：

- `AGENTS.md`
- `contracts/*`
- `docs/catalog.yaml`
- `docs/reference/*`
- `docs/guides/contributing.md`
- `docs/definition-of-done.md`
- 当前 `openspec/changes/<change>/`

这些内容的用途分别是：

- `AGENTS.md`：快速导航
- `contracts/*`：事实源
- `catalog.yaml`：文档影响路由规则
- `reference/*`：人类可读的契约投影
- `definition-of-done.md`：完成标准
- `change`：本次任务上下文

### 10.2 第二阶段应预留的上下文组装机制

后续可以在运行时做：

- 根据当前 stage 和当前 change，自动选择上下文文件
- 根据 `catalog.yaml` 的 `links` 扩展关联契约
- 根据 `contract impact` 裁剪文档范围
- 将“事实源 + 变更上下文 + 校验标准”组装为 agent 输入

但第一阶段不实现自动打包，只确保这些资料路径稳定、职责清楚、可被 agent 检索。

### 10.3 面向大模型的文档要求

要提高模型解决问题的准确率，必须保证：

- 事实源唯一
- 文档分层清晰
- 每份文档职责单一
- 标题稳定，路径稳定
- change 上下文与长期契约可关联

Praxis 的职责是把这些资料组织好，而不是把所有知识塞进一个大提示词。

## 11. 实施顺序

### Phase 1

- `setup --docs-contracts <mode>`
- `docs init`
- `docs check`
- `managed-entry` 增加 docs gate
- 中文模板落地

### Phase 2

- `docs plan`
- `docs sync`
- OpenSpec propose / apply / archive 结构化文档影响字段

### Phase 3

- 基于 `catalog.yaml` 的轻量 change impact 规则
- 基于 `links` 的契约关联分析
- 面向 agent 的上下文组装能力

## 12. 最终结论

这套能力不应无脑强装，也不应只停留在目录模板。

推荐策略是：

- 默认 `minimal`
- 支持升级到 `full`
- 第一阶段交付模板、门禁、检查和最小同步能力
- 第二阶段再把文档契约作为 AI 上下文路由层接起来

一句话总结：

`Praxis 不替用户写所有文档；Praxis 负责在用户项目里安装文档契约结构、定义更新时机、提供校验入口，并把这些稳定资料组织成 AI 可用的上下文。`
