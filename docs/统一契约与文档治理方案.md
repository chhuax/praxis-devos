# 统一契约与文档治理方案

## 1. 背景与目标

很多项目的文档问题，不是“文档不够多”，而是“事实源不清楚”。常见表现包括：

- `docs/` 下存在多套并行 API 文档，内容重复且长期漂移
- 接口说明、架构说明、故障排查、最佳实践混写在同一份文档里
- 变更完成时靠开发者手工判断是否需要补文档
- 文档基数为 0 的项目没有统一起步方式

本方案的目标不是单纯重构 `docs/` 目录，而是建立一套可执行的统一契约模型：

- 对外承诺先落为契约
- 参考文档从契约生成或半生成
- Guide 只承载使用说明和场景化解释
- OpenSpec 变更流程负责收集 impact 并在收尾前做同步检查

核心原则：

- `docs/` 不是事实源，`contracts/` 才是事实源
- 契约和文档必须分层
- 文档同步应纳入 change gate，而不是依赖人记忆

## 2. 适用范围

该方案适用于两类项目：

- 重文档项目：已有大量 Markdown 文档，但内容分散、重复、过时
- 0 文档项目：还没有成体系文档，需要从最小可行模型开始

## 3. 统一契约模型

不同项目的“对外承诺”不同，因此不能只盯着 HTTP API。建议按契约类型分层建模，按需启用。

### 3.1 契约类型

- HTTP API 契约
  - 目录：`contracts/openapi/*.yaml`
  - 用途：REST 接口的路径、参数、响应、错误、鉴权
- 事件契约
  - 目录：`contracts/events/*.yaml` 或 `contracts/events/*.schema.json`
  - 用途：Webhook、消息队列、异步通知
- 配置契约
  - 目录：`contracts/config/*.schema.json`
  - 用途：环境变量、配置项、默认值、取值范围
- 错误码契约
  - 目录：`contracts/errors/errors.yaml`
  - 用途：错误码、状态码、message key、语义说明
- 权限契约
  - 目录：`contracts/permissions/*.yaml`
  - 用途：权限点、角色、资源范围、授权模型
- CLI 契约
  - 目录：`contracts/cli/*.yaml`
  - 用途：命令、参数、输出格式、退出码
- 数据契约
  - 目录：`contracts/data/*`
  - 用途：导入导出数据结构、外部消费数据模型

### 3.2 原则

- 一个对外能力只能有一个主契约
- 参考文档不得作为唯一事实源
- Guide 不重复维护参数表和字段说明
- 变更先改契约，再改实现，再生成参考文档

## 4. 建议目录结构

```text
contracts/
  openapi/
  events/
  config/
  errors/
  permissions/
  cli/

docs/
  reference/
  guides/
  adr/
  changelog/
  catalog.yaml

openspec/
  changes/<change>/
```

### 4.1 各层职责

- `contracts/`
  - 系统对外承诺的唯一事实源
- `docs/reference/`
  - 从契约生成或半生成的参考文档
- `docs/guides/`
  - 手写的使用说明、接入步骤、FAQ、迁移说明
- `docs/adr/`
  - 设计决策与取舍
- `docs/changelog/`
  - 面向用户的变化记录
- `openspec/changes/<change>/`
  - 变更期产物；收尾后应同步回正式契约与正式文档

## 5. 文档分层规则

### 5.1 Reference

Reference 只回答“系统承诺了什么”和“如何调用”，应包含：

- 路径或命令
- 请求参数
- 响应结构
- 错误码
- 鉴权与权限要求
- 示例请求与响应
- 版本与兼容性说明

Reference 不应承担：

- 大段架构背景
- 详细设计过程
- 故障排查手册
- 项目介绍型内容

### 5.2 Guide

Guide 只回答“在什么场景下怎么用”，应包含：

- 典型接入路径
- 操作步骤
- 约束和前置条件
- 升级或迁移说明
- 常见误用与注意事项

Guide 不应重复：

- 完整参数表
- 全量错误码
- 与契约重复的字段定义

### 5.3 ADR

ADR 只回答“为什么这么设计”，应包含：

- 决策背景
- 方案对比
- 最终选择
- 风险与后果

## 6. `docs/catalog.yaml` 设计

`docs/catalog.yaml` 是治理层配置，不是给最终读者看的。它的职责是把“代码变化”映射到“契约与文档更新义务”。

建议最小结构如下：

```yaml
artifacts:
  - id: public-http-api
    kind: api-reference
    contract: contracts/openapi/public.yaml
    outputs:
      - docs/reference/http-api.md
    triggers:
      - controller
      - request-dto
      - response-dto
      - error-code
      - auth-scope
    requireGuideWhen:
      - breaking-change
      - onboarding-flow-change

  - id: app-config
    kind: config-reference
    contract: contracts/config/app.schema.json
    outputs:
      - docs/reference/config.md
    triggers:
      - application-properties
      - env-var
      - config-binding

  - id: permissions
    kind: permission-reference
    contract: contracts/permissions/app.yaml
    outputs:
      - docs/reference/permissions.md
    triggers:
      - auth-annotation
      - permission-code
      - role-binding
```

### 6.1 关键字段建议

- `id`
  - 文档资产唯一标识
- `kind`
  - 文档/契约类型
- `contract`
  - 主契约路径
- `outputs`
  - 受该契约驱动的参考文档输出
- `triggers`
  - 哪些代码变化会触发更新义务
- `requireGuideWhen`
  - 什么场景下还必须补 `docs/guides`

## 7. OpenSpec 集成方案

文档更新不应只是“任务结束后顺便做一下”，而应成为 OpenSpec change 的标准 gate。

### 7.1 `opsx-propose`

在 proposal 阶段新增契约影响采集：

- 是否影响对外契约
- 影响哪些契约类型
- 是否存在 breaking change
- 是否需要 migration guide
- 是否需要更新错误码、权限、配置说明

建议在 change 产物中显式记录：

- `contract impact`
- `doc impact`
- `breaking change`
- `migration required`

### 7.2 `opsx-apply`

在任务阶段新增文档相关任务模板：

- 更新契约文件
- 生成或同步 `docs/reference`
- 补充 `docs/guides` 或迁移说明
- 若无影响，记录 `no-doc-impact` 原因

建议规则：

- 改 controller / request / response / error code / auth 时，默认触发契约检查
- Guide 不是默认必改，只有用户接入方式变更时才要求

### 7.3 `opsx-archive`

在收尾阶段增加 `docs sync assessment`：

- 契约是否已更新
- 参考文档是否已同步
- breaking change 是否已有迁移说明
- 是否存在未完成但被豁免的文档项

归档前给出明确结论：

- `contracts synced`
- `reference synced`
- `guide required and completed`
- `archive with accepted doc risk`

## 8. Praxis DevOS 通用能力建议

如果将该能力沉淀到 `praxis-devos`，建议新增以下命令与行为。

### 8.1 CLI 命令

- `praxis-devos contracts init`
  - 初始化最小契约目录骨架
- `praxis-devos docs init`
  - 初始化 `docs/reference`、`docs/guides`、`docs/catalog.yaml`
- `praxis-devos docs plan --change <name>`
  - 根据 change 输出本次应更新的契约和文档清单
- `praxis-devos docs check --change <name>`
  - 检查契约/文档影响声明和同步状态
- `praxis-devos docs sync`
  - 从契约生成或刷新参考文档

### 8.2 托管规则

在项目托管入口中增加 `Documentation/Contract Gate`，要求：

- 对外接口、配置、权限、CLI、错误码变化必须更新对应契约
- Guide 只在用户使用路径变化时要求补充
- 归档或完成前必须通过 `docs check`，否则需要显式豁免

## 9. 两类项目的落地路径

### 9.1 0 文档项目

目标不是一次性补齐所有文档，而是先建立最小契约闭环。

推荐起步集：

```text
contracts/
  openapi/public.yaml
  errors/errors.yaml

docs/
  reference/
  guides/
  catalog.yaml
```

推荐顺序：

1. 识别最核心的对外承诺
2. 建立最小契约
3. 生成第一版 reference
4. 只补 1 到 2 篇必要 guide
5. 将文档同步纳入 change gate

### 9.2 重文档项目

目标不是删除旧文档，而是先建立主从关系，再逐步收口。

推荐顺序：

1. 盘点现有文档资产
2. 标出重复和冲突的 API 文档
3. 确定唯一 reference 目录
4. 为核心外部能力建立契约源
5. 旧 Markdown 文档过渡为 guide 或被替换
6. OpenSpec 收尾时强制执行 docs check

## 10. 对 `iuap-yms-manage` 的具体判断

对 `/Volumes/MacData/workspace/iuap-yms-manage` 的抽样结果表明：

- `docs` 目录体量大，属于重文档项目
- 存在两套并行 API 文档：
  - `docs/API接口文档/`
  - `docs/API参考手册/`
- 部分文档将接口定义、架构图、实现解释、故障排查混写在一起
- 当前没有清晰可见的统一 OpenAPI/Swagger 契约源
- `openspec/changes/add-config-compare-api/api-doc.md` 表明项目已经有“变更级接口说明”的意识，但尚未与正式契约体系打通

因此该项目的优先级应为：

1. 先合并并收敛 API 文档语义，避免长期两套 reference 并存
2. 为核心外部域建立契约源
3. 将 change 中的 `api-doc.md` 类产物沉淀回正式契约或 reference
4. 在 OpenSpec 收尾前增加 docs/contract check

## 11. 迁移原则

迁移过程中建议遵循以下原则：

- 不要求一次性重写全部旧文档
- 新变更从新规则开始执行，旧文档逐步收敛
- 优先治理高频变化、高外部依赖、高误用风险的接口
- 优先治理对外承诺，而不是内部实现细节
- 保留迁移期兼容文档，但要明确“主文档”与“过渡文档”

## 12. 建议的第一阶段 MVP

如果需要先做可落地的第一版，建议只做以下四项：

1. 建立 `contracts/` 最小骨架
2. 新增 `docs/catalog.yaml`
3. 新增 `praxis-devos docs check`
4. 在 `opsx-propose`、`opsx-apply`、`opsx-archive` 中接入文档与契约检查

这样可以先解决最关键的问题：

- 变化有没有影响外部承诺
- 影响了哪些契约和文档
- 收尾前有没有同步

## 13. 最终结论

正确方向不是“要求团队多写文档”，而是建立下面这条链路：

`变更 -> 契约更新 -> 参考文档生成/同步 -> Guide 补充 -> 收尾检查`

文档治理的核心不在 `docs/` 目录本身，而在于：

- 是否定义了唯一契约
- 是否把参考文档和解释文档分层
- 是否把同步动作接入变更流程

只有这样，0 文档项目能有统一起步方式，重文档项目也能逐步从“手写文档堆积”收敛到“契约驱动文档”。
